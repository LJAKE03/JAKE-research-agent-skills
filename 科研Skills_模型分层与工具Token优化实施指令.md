# 科研 Skills 模型分层与工具 Token 优化实施指令

> **执行对象：** Codex  
> **目标仓库：** 当前 `research-agent-skills-v2` 科研 Skills 项目  
> **执行方式：** 先审计，再制定方案，随后直接修改、测试和验收  
> **版本控制：** 本轮禁止自动 `git commit` 和 `git push`

---

## 1. 总体任务

请对当前 `research-agent-skills-v2` 科研 Skills 项目进行一次完整优化。

项目根目录以当前实际仓库位置为准，不要假设固定盘符。先定位仓库根目录并读取现有文件。

当前启动器、自检、新建项目、已有项目补齐、配置恢复、中文路径及空格路径处理已经通过测试。本次修改不得破坏这些功能。

本轮优化包含两个核心目标：

1. 按任务复杂度自动选择高能力模型或低成本模型；
2. 优化 Python、PowerShell、原生命令及 Codex 内置工具的选择策略，减少 Token、终端输出、重复调用和无效重试。

请先审计，再制定修改方案，确认影响范围后直接实施。不要只给建议，必须修改实际文件、补充测试并运行验证。

## 3. 模型分层总体架构

目标架构为：

- **高能力主代理：** 负责整体规划、任务拆解、技术路线、重大判断和最终审核；
- **中等成本执行代理：** 负责需要一定理解能力、但边界明确的科研支持任务；
- **低成本输出代理：** 负责格式化、重复性、机械性和低风险任务；
- 所有最终结论仍由高能力主代理审核并输出。

不得让低成本代理独立决定：

- 总体研究方向；
- 项目任务拆解；
- 技术路线；
- 创新点；
- 数值模型选择；
- 仿真参数；
- 边界条件；
- GT-POWER 标定方案；
- 下一批 Case 设置；
- Fluent 发散原因；
- 实验方案；
- 工程安全问题；
- 论文核心结论；
- 文献可靠性；
- 公式、单位和数值是否正确。

## 4. 高能力主代理职责

将当前账户中可用的最高能力模型配置为科研 Agent 的主代理模型。

用户期望的逻辑名称为：

| 逻辑层 | 预期用途 |
|---|---|
| `strategic / sol` | 最高能力层 |
| `support / terra` | 能力与成本均衡层 |
| `economy / luna` | 低成本输出层 |

实际模型 ID 必须来自本机查询结果。

如果本机存在对应模型，则建立映射，例如：

- `strategic_model` = 当前可用的 Sol 或最高能力模型；
- `support_model` = 当前可用的 Terra 或中等成本模型；
- `economy_model` = 当前可用的 Luna 或最低成本模型。

如果没有 Sol、Terra 或 Luna 的正式模型 ID，不得编造。应按以下原则回退：

1. `strategic`：使用当前账户可用的最高能力模型；
2. `support`：使用次一级或成本较低的可用模型；
3. `economy`：使用最低成本的可用模型；
4. 如果只能使用一个模型，则通过 reasoning effort、verbosity、任务上下文和子代理权限实现分层；
5. 将实际映射和回退原因写入说明文档。

高能力主代理必须负责：

- 识别用户真实目标；
- 制定整体研究计划；
- 拆分阶段、任务和里程碑；
- 确定任务依赖关系；
- 决定调用哪些子代理；
- 处理子代理之间的冲突；
- 审核来源、数据、单位和结论；
- 对输出进行最终验收；
- 生成最终面向用户的回答。

对于以下任务，主代理默认使用较高 reasoning effort：

- 科研项目启动；
- 整体规划；
- 技术路线设计；
- 研究空白与创新点分析；
- 多物理场建模方案；
- 仿真标定；
- 工程异常分析；
- 论文结构与核心论证；
- 关键数据解释；
- 最终结论和决策。

## 5. 支持型子代理

如果当前 Codex 版本支持自定义 agents/subagents，请建立支持型科研代理，例如：

```text
.codex/agents/research-support.toml
```

或当前版本要求的等效位置。

该代理优先使用 `support/terra` 层模型，适合：

- 扫描项目目录；
- 阅读大量文件；
- 从实验报告中提取参数；
- 从文献中提取题名、方法、工况和结果；
- 汇总已有仿真结果；
- 按主代理已经确定的指标进行初步排序；
- 检查缺失文件；
- 整理候选方案；
- 生成供主代理审核的中间材料；
- 执行边界明确的测试和脚本。

支持型代理不得：

- 擅自改变研究路线；
- 自行确定核心参数；
- 自行生成缺失数据；
- 将候选判断写成确定结论；
- 修改核心配置而不报告；
- 直接向用户发布未经主代理审核的最终科研结论。

其返回结果应包含：

- 完成了什么；
- 使用了哪些文件或数据；
- 提取了哪些关键结果；
- 存在哪些异常；
- 哪些内容仍不确定；
- 建议主代理重点复核什么。

## 6. 低成本输出代理

如果当前 Codex 版本支持，建立低成本输出代理，例如：

```text
.codex/agents/research-output.toml
```

或当前版本要求的等效位置。

该代理优先使用 `economy/luna` 层模型，只处理同时满足以下条件的任务：

1. 任务目标已经确定；
2. 输入内容完整；
3. 输出格式明确；
4. 不需要新的科研判断；
5. 结果能够机械检查；
6. 出错不会直接改变核心研究结论；
7. 主代理能够低成本抽查。

适合任务包括：

- Markdown、TXT、CSV、JSON 等格式转换；
- 表格字段重排；
- 模板填充；
- 文件清单生成；
- 标题和编号统一；
- 标点、空格和格式规范化；
- 已确认文字的压缩或结构化；
- 已确认内容的 PPT 化整理；
- 参考文献格式的机械统一；
- 批量重命名非关键文件；
- 生成辅助性目录和索引；
- 将主代理已经确认的内容转成指定格式。

低成本代理禁止处理：

- 总体规划；
- 任务拆解；
- 技术方案选择；
- 复杂科研解释；
- 公式推导；
- 数值判断；
- 模型标定；
- 异常诊断；
- 研究创新性判断；
- 文献真实性核验；
- 最终结论；
- 需要高可靠性的正式投稿文本；
- 工程安全相关判断。

任何不确定信息必须明确标记为“待主代理确认”，不得自行补齐。

## 7. 模型路由规则

在科研 Skill 总控文件中加入明确、可执行的路由规则。

### 7.1 A 类：战略与关键判断任务

- 必须由 `strategic` 主代理执行；
- 默认使用 `high` 或 `xhigh` reasoning；
- 禁止委派最终责任。

### 7.2 B 类：科研支持任务

- 可委派给 `support` 代理；
- 主代理必须复核；
- 支持代理只能给候选结论。

### 7.3 C 类：低风险输出任务

- 可委派给 `economy` 代理；
- 主代理执行抽样检查或格式验证；
- 不允许引入新的事实、数据、引用或科研判断。

### 7.4 D 类：混合任务

- 先由主代理完成判断和框架；
- 再将明确的执行部分拆给 `support` 或 `economy`；
- 最后由主代理合并和审核。

### 7.5 路由约束

- 不得为了使用低成本模型而强行拆分极小任务；
- 子代理调用本身有额外上下文成本；
- 只有任务边界明确且能够减少主代理工作量时才委派；
- 默认并发子代理不超过 2 个；
- 子代理不得递归创建子代理；
- 主代理必须阅读子代理输出；
- 子代理冲突时由主代理裁决；
- 最终面向用户的答复由主代理生成；
- 关键结论必须附可验证依据；
- 不得以模型名称代替质量控制。

## 8. 工具选择与 Token 优化

不要建立“所有任务一律优先 Python”的机械规则。

应建立以下工具选择原则：

优先选择能够实现以下目标的工具：

- 最少模型交互轮次；
- 最少终端输出；
- 最低失败率；
- 最少重复读取；
- 最清晰验证结果；
- 最容易复用；
- 最低上下文占用。

建议工具优先级为：

1. Codex 内置的读取、搜索、编辑、补丁和文件工具；
2. 专用原生命令，例如 `git`、`rg`、`rg --files`；
3. 已有且经过验证的项目脚本；
4. Python；
5. PowerShell；
6. 临时生成的大段终端脚本。

该优先级不是绝对规则，必须按任务类型选择。

## 9. Python 使用规则

以下任务优先使用 Python：

- CSV、JSON、Excel 和结构化数据处理；
- 科研计算；
- 数值分析；
- 批量文本处理；
- 多文件解析；
- 数据清洗；
- 统计分析；
- 绘图；
- 自动生成结构化结果；
- 需要跨平台复用的处理；
- 需要单元测试的逻辑；
- 同类操作需要重复执行；
- 一次读取大量数据并输出简洁摘要。

当某类复杂处理预计执行两次以上时：

- 优先生成可复用 `.py` 脚本；
- 不要每次重新生成临时代码；
- 为脚本增加参数；
- 为脚本增加明确的退出码；
- 支持 `--quiet`、`--json` 或等效模式；
- 完整结果写入文件；
- 终端只输出摘要、路径和异常；
- 对关键逻辑补充最小测试。

不得使用 Python 重新实现一条 PowerShell、`git` 或 `rg` 命令即可可靠完成的简单操作。

## 10. PowerShell 使用规则

以下任务优先使用 PowerShell：

- Windows 环境变量；
- 可执行程序定位；
- 快捷方式；
- 注册表；
- Windows 服务和进程；
- Windows 文件属性；
- 启动器；
- 安装与卸载；
- 执行策略；
- 系统路径；
- CMD 与 PowerShell 兼容性；
- PowerShell 5.1 兼容性；
- 简单文件复制、移动和目录创建；
- 调用 Windows 原生工具。

以下任务不要强制改成 Python：

```powershell
Get-Command
Test-Path
Get-ChildItem
Copy-Item
Move-Item
New-Item
Get-FileHash
```

还包括：

- 环境变量读写；
- 快捷方式创建；
- 启动 Codex；
- 启动器自检。

现有启动器及 Windows 集成脚本原则上继续使用 PowerShell/CMD，不要为了形式统一而重写成 Python。

## 11. Codex 内置工具和原生命令规则

优先使用：

- `rg` 搜索文件内容；
- `rg --files` 枚举文件；
- `git status`、`git diff`、`git log` 等 Git 原生命令；
- 精确补丁修改；
- 小范围读取；
- 有目录和文件类型限制的搜索。

禁止：

- 无差别递归打印整个仓库；
- 反复读取同一个大文件；
- 使用 `Get-Content` 或 `cat` 输出完整大型文件；
- 每修改一个字符就运行全量测试；
- 将大段日志全部送入上下文；
- 用 Python 重新实现 Git；
- 用 PowerShell 逐文件循环模拟一次 Python 批处理即可完成的工作。

## 12. 输出控制规则

所有命令和脚本默认减少无关输出。

必须加入以下约束：

- 成功时只输出关键状态；
- 失败时输出错误位置、错误原因和建议动作；
- 大型结果写入文件；
- 终端只显示结果路径、数量、摘要和异常；
- 搜索必须限定目录、关键词和文件类型；
- 表格只显示必要列；
- 日志默认关闭或使用 quiet 模式；
- 调试时才允许详细日志；
- 不得默认打印完整 Prompt、完整配置或完整文件；
- 能使用 JSON 摘要时优先使用结构化摘要；
- 测试输出应有 `PASS`、`WARN`、`FAIL` 和最终退出码。

## 13. 避免重复工作的规则

在 Skill 中增加：

- 优先复用已有脚本；
- 修改前先检查是否已有同类能力；
- 不重复扫描未变化目录；
- 不重复读取已经解析的文件；
- 对大型文件建立轻量缓存或结果文件；
- 修改后只运行相关最小测试；
- 最终验收阶段再执行全量自检；
- 出错时先分析根因，不盲目重复同一命令；
- 连续两次同类失败后必须改变诊断方法；
- 不得无限重试；
- 临时文件应集中放在临时目录并在测试后清理。
